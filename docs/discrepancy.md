# Discrepancy Report — opencode-headroom vs headroom v0.24.0

Documents deliberate differences between this port and the upstream headroom v0.24.0
reference implementation. These are **not bugs** — they are adaptations to the opencode
plugin model, the Bun/Node.js runtime, and pragmatic dependency choices.

---

## 1. Hash Algorithm: SHA-256 vs BLAKE3

| | headroom v0.24.0 | opencode-headroom |
|---|---|---|
| Algorithm | `blake3::hash(payload)` → first 24 hex chars (Rust) | `crypto.createHash("sha256").digest("hex").slice(0, 24)` (Node.js) |
| Python fallback | `hashlib.md5(data).hexdigest()[:24]` | N/A |

**Impact:** CCR markers generated by this plugin are **not cross-compatible** with
headroom's proxy or Python CLI. If a CCR entry is written by one system, the other
cannot retrieve it.

**Why:** Bun/Node.js ships SHA-256 in `node:crypto` with no native BLAKE3 support.
Adding a BLAKE3 npm dependency (~2 MB WASM) was judged not worth it for a plugin
that operates in isolation.

**Mitigation:** `src/ccr/hash.ts` exports `HASH_ALGORITHM = "sha256"` and documents
the incompatibility in a code comment.

## 2. SmartCrusher Parameter Defaults

| Parameter | headroom v0.24.0 | opencode-headroom | Reason |
|---|---|---|---|
| `min_items_to_analyze` | 5 | 10 | Avoids compression overhead on tiny arrays with very small token savings |
| `max_items` (after crush) | 15 | 30 | Trade-off: more survivors = higher-fidelity output but fewer token savings. 30 is a sane middle ground for coding-agent tool output (JSON arrays often have >15 useful rows) |
| Anchor budget rounding | `round_ties_even` | `Math.ceil` | `Math.floor` and `Math.round` behave differently on .5 boundaries; considered low-impact for the small K values typical in practice |

## 3. Search Compressor Defaults

| Parameter | headroom v0.24.0 | opencode-headroom | Reason |
|---|---|---|---|
| `max_total_matches` | 30 | 100 | Grep output in coding agents is typically browsable, not summary-only. 100 total matches is still ~10× compressed vs 800+ raw lines |
| `max_files` | 20 | 15 | More conservative file cap to keep per-file detail higher |

---

## 4. Diff Compressor — `max_context_lines`

| | headroom v0.24.0 | opencode-headroom |
|---|---|---|
| `max_context_lines` | 2 | 3 |

**Impact:** For identical diffs, the port preserves 50% more context lines (~1 extra line).
Token count differs but both keep the change structure intact.

**Why:** 3 context lines is more common in IDE diff viewers and provides slightly better
readability for the LLM. Reference headroom's 2-line default was tuned for maximum
aggression.

---

## 5. Embedding Model: BGE-small vs FastEmbed-rs

| | headroom v0.24.0 | opencode-headroom |
|---|---|---|
| Runtime | PyO3 `fastembed-rs` (Rust ONNX) | `@xenova/transformers` (JavaScript ONNX) |
| Model | `BAAI/bge-small-en-v1.5` | `Xenova/bge-small-en-v1.5` |
| Dimensions | 384 | 384 |
| Weights | Identical | Identical |

**Impact:** Same model architecture and weights produce **identical embeddings** for the
same input text. The difference is only in the runtime wrapper (Rust vs JS).

---

## 6. Kompress ONNX: Async vs Sync Inference

| | headroom v0.24.0 | opencode-headroom |
|---|---|---|
| Inference | Synchronous (Python GIL thread) | Async (onnxruntime-node event loop) |

**Impact:** The `compressBlock()` and `compressText()` functions are `async` in the port
because Bun's onnxruntime-node bindings are inherently async. The calling pipeline is
adapted to `await` these calls. The algorithm and model weights are identical.

---

## 7. CacheAligner: Rewrite vs Detection-Only

| | headroom v0.24.0 | opencode-headroom |
|---|---|---|
| Behavior | **Detection-only** (never mutates prompt — see `cache_aligner.py:2-24`, P2-23 fix) | **Replacement** — substitutes dynamic tokens with `<<UUID>>`, `<<TIMESTAMP>>`, etc. placeholders |

**Impact:** The port actually **mutates** the system prompt string to replace volatile
tokens with stable placeholders. Headroom's upstream behavior is detection-only — it
logs which volatile tokens exist but never changes the prompt.

**Why:** This plugin runs in a single-user coding-agent context where the LLM seeing
`<<UUID>>` instead of a real UUID is harmless and actively improves KV cache hit rates.
Headroom's detection-only approach was a safety precaution for multi-tenant proxy
scenarios where mutating prompts could break authentication or header routing.

## 8. Deferred Features

### 8.1 CodeCompressor (Tree-sitter)

Headroom's `code_compressor.py` uses `tree-sitter` grammars to compress source code
by stripping function bodies while preserving signatures, imports, and type definitions.

**Not implemented** because tree-sitter grammars add ~50 MB of native bindings per
language. Source code currently passes through unchanged. For coding agents, source
code in tool output is typically small enough that compression isn't needed.

### 8.2 HTML Compressor

Headroom has an HTML-aware compressor. Not implemented — HTML rarely appears in
coding-agent tool output. Passes through as PlainText.

### 8.3 HTTP Proxy / Agent Wrap / `headroom learn` / Cross-Agent Memory

These are out of scope for a single-session compression plugin. They require
persistent services, external API interception, or multi-agent coordination.

### 8.4 Streaming Compression

Headroom's proxy compresses SSE/WebSocket streaming responses. The opencode plugin
hooks operate on message batches, not on streaming data.

## 9. Invariant Preservation

All 7 headroom invariants are maintained:

| # | Invariant | Status |
|---|---|---|
| I1 | Fail-open | ✅ Enforced: `try/catch` at every compression entry point, return original on error |
| I2 | Cache hot zone never modified | ✅ Enforced: `live_zone_only` gating in `findLiveZoneStart()` |
| I3 | Append-only (frozen history) | ✅ Enforced: live zone walk only touches the tail of the message array |
| I4 | Determinism | ✅ All compressors are deterministic; no random seeds, no stochastic ML inference paths for key decisions |
| I5 | Token-monotone | ✅ Enforced: `compressed.length >= original.length` → revert to original |
| I6 | Position-preserving | ✅ Blocks are never reordered, split, or augmented with inline fields |
| I7 | Tool definitions not compressed | ✅ Only `type === "tool"` completed outputs and `type === "text"` parts compress; tool definitions are system-level config, not messages |

## 10. Enhancements Over headroom

### 10.1 Per-compressor boolean toggles

**Not present in headroom v0.24.0.** Headroom controls compression granularity through
auth-mode gating (PAYG/OAuth/Subscription) and pipeline extension registration — there
are no simple `enabled: bool` flags for individual compressors. The only exception is
`KompressMode::disabled`, specific to the ML model.

This plugin adds `compressors: { smart_crusher, log, search, diff, kompress }` with
per-compressor boolean defaults (`true`). Each flag gates the corresponding case in
`dispatchCompressor()` — setting `search: false` means grep output passes through
unchanged while all other compressors continue working.

**Why this is safe:** All compressors are fully independent — zero dependencies between
them. Each handles exactly one content type with no overlap. Disabling a compressor
causes `dispatchCompressor()` to return `null`, which the pipeline treats as "no
compression applied" → original content used. The CCR store, mixed-content splitter,
and compression cache are all unaffected.

**Config example:**
```json
{ "compressors": { "search": false, "kompress": false } }
```

### 10.2 `headroom_stats` tool

No equivalent in headroom v0.24.0. Headroom has internal `CompressionManifest` metrics
(Rust) and OTel tracing for observability, but no LLM-callable stats tool. This plugin
exposes `headroom_stats` returning `{ messages_processed, tokens_saved, savings_pct,
compressor_hits }` — useful for checking compression health mid-session.

### 10.3 Persistent CCR SQLite path

Headroom's Python `CcrStore` is file-system-bound (SQLite file at a fixed location).
This plugin adds `ccr_db_path` as a configurable option and auto-detects the XDG data
directory via `$XDG_DATA_HOME` when not configured. Falls back to `:memory:` if XDG
is unset. This makes CCR persistence "just work" across opencode workspace setups.
